# Smart Correlation Engine

## What It Is

A Smart Correlation Engine matches the current incident against past incidents using multiple dimensions — not just keyword search. Instead of `grep -i "{keywords}" knowledge-base.md`, it correlates by affected service, error signature, bug category, contributing factors, affected code paths, and temporal patterns. A keyword search for "null reference" returns noise. A multi-dimensional correlation for "null reference + payment service + data flow bug + last seen 3 weeks ago" returns the incident that was probably the same underlying issue.

The correlation engine serves two purposes:
1. **Speed**: If this incident was solved before, skip investigation and apply the known fix
2. **Pattern detection**: If this incident KEEPS happening, the previous fix was insufficient — escalate to systemic remediation, not another point fix

Enterprise incident management systems (PagerDuty, ServiceNow, Jira Service Management) all implement correlation as a core feature because the fastest resolution is recognizing that you've seen this before.

## Enterprise Origin

**Source:** PagerDuty Past Incidents, ServiceNow Event Correlation, Microsoft Sentinel incident correlation and matching analytics.

PagerDuty's Past Incidents feature: *"Past Incidents allow Responders to view past resolved Incidents that have similar metadata and were generated on the same Service as their current active Incident. Past Incidents add helpful context for accurate triage, which can lead to shorter resolution time. Responders can see who was involved in a previous Incident, when these types of Incidents happened, and dive into Incident details to discover the remediation steps that were taken."*

The key phrases: "similar metadata," "same Service," "who was involved," "when these types happened." PagerDuty doesn't just match keywords — it matches across structured dimensions: service, metadata, time, and responder history.

PagerDuty also implements alert aggregation: *"Multiple alerts can be aggregated into a single incident for triage, which streamlines incident handoff between teams, centralizes critical information, and reduces notification fatigue."* This is correlation in real-time: recognizing that multiple alerts are symptoms of the same underlying incident.

Microsoft Sentinel's matching analytics: *"Triage an incident generated by matching analytics"* — Sentinel correlates new incidents against existing patterns using automated rules. When a new incident matches the pattern of a known incident type, it's tagged and routed accordingly, skipping manual triage.

## How It Works

### 1. Multi-Dimensional Matching (replace keyword grep)

When checking the knowledge base, match on these dimensions with weighted scoring:

| Dimension | Weight | Match Criteria | Example |
|-----------|--------|---------------|---------|
| **Error signature** | High | Same error message, exception type, or error code | "NullReferenceException in PaymentService.Charge" |
| **Affected service/module** | High | Same file, module, or service affected | `src/payment/charge.ts` |
| **Bug category** | Medium | Same category (logic, boundary, data flow, etc.) | "Data flow" |
| **Contributing factors** | Medium | Same contributing factors identified in postmortem | "Missing null check on API response" |
| **Code path** | Medium | Same function or call chain involved | `checkout → payment → charge → processResponse` |
| **Temporal pattern** | Low | Occurs at similar times (end of month, after deploys) | "Happens after monthly batch job" |
| **Environment** | Low | Same environment-specific conditions | "Only in production with high load" |

### 2. Correlation Scoring

For each past incident, calculate a correlation score:

```
score = 0
if error_signature matches: score += 30
if same service/module:      score += 25
if same bug category:        score += 15
if same contributing factors: score += 15
if same code path:           score += 10
if temporal pattern matches: score += 5

if score >= 50: "HIGH correlation — likely the same issue"
if score >= 30: "MEDIUM correlation — possibly related"
if score <  30: "LOW correlation — probably different"
```

### 3. Correlation Actions

| Correlation | Action |
|------------|--------|
| **HIGH (>= 50)** | "This matches a previous incident: {title} ({date}). The fix was: {resolution}. Apply the same fix? [Y/n/investigate]" |
| **MEDIUM (30-49)** | "This may be related to a previous incident: {title}. The root cause was: {cause}. Check this path first? [Y/n]" |
| **LOW (< 30)** | Proceed with standard investigation. No match surfaced. |

### 4. Recurrence Detection

When a HIGH correlation match is found AND the previous incident was already resolved:

"This is the {N}th occurrence of this issue pattern:
- {date}: {title} — fixed with {resolution}
- {date}: {title} — fixed with {resolution}
- NOW: same pattern recurring

The previous fixes were INSUFFICIENT. This needs systemic remediation,
not another point fix. Consider:
- Adding a guardrail (test, lint rule, CI check) that prevents this class of bug
- Addressing the underlying architectural issue that makes this failure mode possible
- Escalating to tech debt remediation if the root cause is known but unfixed"

### 5. Knowledge Base Structure for Correlation

To support multi-dimensional matching, knowledge base entries need structured fields:

```markdown
### {Issue Title} ({date})
**Category:** {bug type}
**Severity:** SEV{N}
**Service:** {affected service/module}
**Error signature:** {exact error message or exception}
**Code path:** {function call chain}
**Symptom:** {what the user saw}
**Root Cause:** {what was actually wrong}
**Contributing Factors:** {list}
**Fix:** {what resolved it}
**Guardrails added:** {what was added to prevent recurrence}
**Recurrence count:** {how many times this pattern has occurred}
**Related incidents:** {links to correlated past incidents}
**Plan:** {plan name if linked}
**Log:** {path to full troubleshooting log}
```

## Why It Prevents Gaps

- **Finds matches that keyword search misses.** "Payment timeout" and "charge gateway connection refused" are different keywords but the same class of bug. Multi-dimensional matching catches this.
- **Detects recurrence that point fixes don't solve.** If the same pattern appears 3 times with 3 different point fixes, the correlation engine flags it as a systemic issue.
- **Speeds up investigation.** A HIGH correlation match with a known fix can resolve an incident in minutes instead of hours.
- **Surfaces related incidents for postmortems.** The "Related Incidents" section of a postmortem is populated automatically by correlation, not manually by memory.
- **Builds institutional memory that scales.** After 50 incidents, keyword search is unusable. Multi-dimensional correlation remains useful because it filters by structured dimensions.

## Status Before Implementation

Our knowledge base search is `grep -i "{issue-keywords}" knowledge-base.md`. This:
- Matches on keywords only, missing semantically similar but lexically different issues
- Returns all matches with no relevance ranking
- Doesn't correlate by service, category, or code path
- Doesn't detect recurrence patterns
- Doesn't link related incidents

The knowledge base entries are flat markdown with unstructured fields, making programmatic correlation impossible.

## Implementation

- Add structured fields to knowledge base entries (service, error signature, code path, contributing factors, recurrence count)
- Replace keyword grep with multi-dimensional matching logic
- Add correlation scoring with weighted dimensions
- Add recurrence detection and systemic remediation escalation
- Add related incident linking (bidirectional)
- Add correlation score to troubleshooting output: "Correlation check: {HIGH/MEDIUM/LOW} match with {title}"
